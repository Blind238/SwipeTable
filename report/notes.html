Thought about pagination vs endless scrolling.

Case for pagination:

  • Harder to have a smooth experience when scrolling fast while endless.
  • Easier to see and feel your place in the data.

Case for endless scrolling:

  • If user doesn't know what they're looking for and want to skim.
  • Hitting onscreen buttons doesn't seem like fun if browsing a large dataset.

Thought about sorting and filtering.

Filtering should be available on mobile due to the smaller screen space. Sorting should be available to all.

Due to the focus on performance, the decision was made to let the server handle filtering and sorting.

When you have received a dataset from the server, then you sort it by another column, it does not sort ALL of the relevant data. It sorts whatever data it has received. Their are a couple of ways to solve this:

  • When there's a sorting action, request another dataset with the relevant sorting, and when the data is received it should be displayed and the sorting noted and displayed.
  • Let the client receive all of the data and do the sorting there. The problem with this is that the amount of data the client has to handle might be enough to cause noticeable performance degradation.

The solution that should be used can depend on the device used. The safest way is to let the server do the sorting and resend the data.

Another thing that just occurred to me is that a log is a living set of data, changing constantly. How do you make sure that the next page of data is not moved because of new data? Imagine a new item comes on the list and you request the next page. Since services should be stateless the server will not know what you have previously requested and may send duplicate data.

At first I thought I'd have sorting by value on the server side. However, sorting by X would be better. I don't know if I even need to worry about how sorting is handled on the server side.

On the issue of keeping place in the document, each request beyond the first should also send the last ID received of that same type of request. It's a simple way to avoid duplicates, but what if a newer item appears in the middle of that because of the sorting?

Perhaps also send a timestamp of last request. it would make it possible for the server to have one query before the timestamp and one after to count and to present it as new data. Perhaps only the former, and the plugin could poll for new items every second? It would depend on the amount of new items and UX.

Filtering seems to be another issue entirely. Filter on search results? Filter on available values (dynamic)?

The more I think about it, the more I feel like having a dumb client is the best option. I don't know the specifics of the device yet but if the client is dumb it doesn't matter.

Only performance hit is on waiting for new data, no waiting for the client to sort or doing anything with it. Should I start thinking about server infrastucture? I can't know or ask about that until Egor is back, might as well keep going with the Javascript.

Performance

In regards to DOM manipulation, to lower the memory footprint I have to reuse exisiting DOM elements. With pagination this means replacing the contents. With infinite scrolling this means finding a way to store a reference to all DOM elements and reuse those outside of the view.

Interface

I'm thinking that tapping on a column can sort by that column, with feedback as arrows. To go to next page you can swipe. I feel that this allows for easier Javascript DOM manipulation and reuse, while making it easy to implement rapid paging. If the paging is fast enough (because of CSS) there's no need to have infinite scrolling. Need methods for actions such as refreshing and removing(completing?) a log. Pull down to get to first page with fresh data. Also, by placing a constraint like a page, the user shouldn't see a blank page while data is being fetched, which is more likely with infinite scrolling.

I think for portrait vs landscape:

  • Page navigation via the shortest route. So in landscape it would be from bottom to top, in portrait it would be right to left. Swiping right to left while in landscape I feel is too slow. It's a lot of energy to expend for just the next set of data.
  • Portrait view and landscape view should both implement a set of important columns to be displayed. The problem with portrait view and important columns is that navigating the columns might trigger a page swipe. Detection areas should probably be iterated on. Perhaps moving on the columns navigates between the columns. This calls for paper prototyping to see user expectation. It should be a loose test to explore how users expect the system to behave.
  • If the test data does not show a winning way to navigate, all the options that were present will be considered as configurable options for the plugin.

On any page, the next page should be fetched and populated as soon as possible.

While writing the flow for switching pages, I started thinking about the swiping behavior. It might pose problems if it's an android system. Swiping from the bottom could trigger the navigation bar, if it's a buttonless system. It could also trigger Google Now if it is running Jellybean. I still think it's the better way (pending user testing). I believe I need to have the preferred behavior as default, and give options to the implementer of the plugin.
